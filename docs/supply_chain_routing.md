# Система управления цепочками поставок

## Цели системы
- Оптимизация маршрутов доставки сырья и готовой продукции.
- Снижение транспортных расходов.
- Минимизация потерь продукта при транспортировке и хранении.

## Архитектура решения
1. **Модуль интеграции данных**
   - Источники: ERP, WMS, TMS, геоинформационные системы.
   - Функции: нормализация данных, обогащение координатами и временными отметками.
2. **Модуль моделирования сети поставок**
   - Представление сети в виде ориентированного графа: узлы (заводы, месторождения, склады), ребра (участки дорог, маршруты).
   - Атрибуты узлов: производительность, запас, окна загрузки/выгрузки, риски.
   - Атрибуты ребер: длина, тип дороги, покрытие, ограничение скорости, текущая загруженность, прогноз дорожных условий.
3. **Модуль прогнозирования**
   - Использует исторические данные телематики и открытые источники для предсказания состояния дорог, трафика, погодных факторов.
   - Выдает временные функции стоимости для ребер графа.
4. **Оптимизационный модуль**
   - Алгоритмы: многокритериальное кратчайшее дерево, стохастическая оптимизация, методы линейного программирования для распределения потоков.
   - Включает механизм пересчета маршрутов в реальном времени при изменении условий.
5. **Модуль мониторинга и оповещений**
   - Визуализация маршрутов, KPI (стоимость, время, потери).
   - Автоматические оповещения о нарушении SLA, задержках, авариях.

## Модель данных
- **Географические данные**: координаты узлов, категории дорог, высотность, климатические зоны.
- **Транспортные средства**: грузоподъемность, расход топлива, стоимость километра, доступность.
- **Ограничения**: окна доставки, грузовые ограничения дорог, требования к температурному режиму.
- **Риски и потери**: вероятность повреждения груза, потерь при перегрузке, сезонные риски.

## Алгоритм выбора оптимального маршрута

### Постановка задачи
Найти маршрут доставки от месторождения к заводу, минимизирующий суммарные расходы и потери с учетом дорожных условий и загрузки транспорта.

### Основные этапы
1. **Построение графа**
   - Узлы: месторождения, заводы, склады, перегрузочные станции.
   - Ребра: возможные дорожные участки и виды транспорта.
   - Вес ребра `w_{ij}` формируется как комбинированная стоимость: 
     \[
     w_{ij} = \alpha \cdot C_{\text{топливо}} + \beta \cdot C_{\text{время}} + \gamma \cdot C_{\text{потери}} + \delta \cdot C_{\text{риска}}
     \]
     где коэффициенты \(\alpha, \beta, \gamma, \delta\) задаются бизнес-приоритетами.
2. **Оценка состояния ребер**
   - Время и стоимость проходят через прогнозирующую модель: `C_время` учитывает заторы, ограничения скорости, погодные условия.
   - `C_потери` отражает вероятность порчи груза (например, из-за плохого покрытия) и стоимость страхования.
   - `C_риска` представляет вероятность форс-мажоров (оползни, ремонт дорог) и штрафы за опоздание.
3. **Учет ограничений**
   - Наложение ограничений на грузоподъемность: исключение ребер, где транспорт не проходит.
   - Учет окон доставки: добавление временных ограничений с помощью методов тайм-слотов.
   - Фильтрация по типу продукта (температурный режим, опасность).
4. **Оптимизационный расчет**
   - Используем модифицированный алгоритм Дейкстры/А* для многокритериальной оптимизации.
   - Функция стоимости: 
     \[
     f(n) = g(n) + h(n)
     \]
     где `g(n)` — накопленная стоимость до узла `n`, `h(n)` — эвристика (например, оставшееся расстояние, скорректированное прогнозом).
   - Для учета динамики используется алгоритм *Time-Dependent Shortest Path* (TDSP) с обновляемыми весами по временным интервалам.
5. **Перебалансировка и сценарный анализ**
   - При изменении условий (ДТП, погода) выполняется пересчет маршрутов на ограниченном подграфе.
   - Для глобальной оптимизации потоков используется модель линейного программирования: 
     \[
     \min \sum_{(i,j)} w_{ij} x_{ij}
     \]
     при ограничениях на потоки, мощности узлов и доступность транспорта.

### Псевдокод TDSP-А*
```pseudo
function TDSP_A*(graph, start, goal, departure_time):
    open_set <- priority queue
    g_score[start] <- 0
    h_score[start] <- heuristic(start, goal, departure_time)
    enqueue(open_set, start, g_score[start] + h_score[start])

    while open_set not empty:
        current <- dequeue(open_set)
        if current == goal:
            return reconstruct_path(current)

        for edge in outgoing_edges(current):
            travel_time <- edge.travel_time(departure_time)
            travel_cost <- edge.travel_cost(departure_time)
            tentative_g <- g_score[current] + travel_cost
            arrival_time <- departure_time + travel_time

            if tentative_g < g_score[edge.target]:
                g_score[edge.target] <- tentative_g
                came_from[edge.target] <- current
                priority <- tentative_g + heuristic(edge.target, goal, arrival_time)
                enqueue(open_set, edge.target, priority)

        departure_time <- update_time(current, departure_time)

    return failure
```

### Результаты и KPI
- Сокращение логистических издержек на 10–15% за счет выбора оптимальных маршрутов.
- Снижение времени доставки и повышение соблюдения окон доставки.
- Минимизация потерь продукта за счет учета рисков и состояния дорог.

## Внедрение и интеграция
- Этап 1: Сбор и очистка данных, подключение телематики.
- Этап 2: Построение цифрового двойника сети, верификация модели.
- Этап 3: Внедрение TDSP-А* в пилотном регионе, оценка KPI.
- Этап 4: Масштабирование на всю сеть, добавление сценарного моделирования и симуляций.

## Дополнительные улучшения
- Использование машинного обучения для прогнозов спроса и динамического планирования производства.
- Автоматизированное распределение транспорта с учетом графиков ТО.
- Интеграция с системами контроля качества для отслеживания состояния продукта.

## Производственная готовность прототипа

- **Валидация данных**: модуль `supply_chain.config` гарантирует корректность входных сценариев, проверяя наличие узлов, допустимость
  значений и необходимость явного подтверждения повышенных рисков.
- **Ошибки и логирование**: единая иерархия исключений и настраиваемый логгер CLI обеспечивают прозрачную диагностику проблем при
  запуске.
- **Форматы вывода**: консольное приложение умеет отдавать результат в текстовом и JSON видах, что упрощает интеграцию с другими
  сервисами и пайплайнами.
- **Покрытие тестами**: набор pytest тестов проверяет корректность маршрута на эталонном сценарии и правила валидации конфигураций.
